## 3. Confirmed Issues

### [P1] PG 限流故障时生产环境退回单进程内存限流

- Location: [`src/ideago/api/rate_limit.py:144`](/Users/simonsun/github_project/IdeaGo/src/ideago/api/rate_limit.py:144), [`src/ideago/api/rate_limit.py:201`](/Users/simonsun/github_project/IdeaGo/src/ideago/api/rate_limit.py:201)
- Impact: hosted 多实例环境下，Supabase RPC 短暂故障会让每个实例各自计算限流窗口；攻击者或高频用户可以通过负载均衡获得高于计划的请求量，影响成本和公平性。
- Evidence: `_check_rate_limit_pg()` 在非 200、timeout、HTTPError 和 unexpected exception 后都会返回 `_check_rate_limit_memory(...)`，没有区分 production 和 development，也没有向调用方暴露 degraded 状态。
- Recommendation: 生产环境改为 fail-closed 或按更保守的短窗口内存限流，同时增加 `rate_limit_degraded` 指标/告警；本地开发可继续保留内存 fallback。

### [P2] 报告状态路由有未统一处理的依赖失败路径

- Location: [`src/ideago/api/routes/reports.py:204`](/Users/simonsun/github_project/IdeaGo/src/ideago/api/routes/reports.py:204), [`src/ideago/api/routes/reports.py:216`](/Users/simonsun/github_project/IdeaGo/src/ideago/api/routes/reports.py:216)
- Impact: 当 Supabase reports/status 在 owner 校验之后抖动，用户可能拿到通用 500，而不是前端可识别的 `DEPENDENCY_UNAVAILABLE`；这会削弱报告恢复和运维诊断。
- Evidence: `get_report_status()` 只把 owner 查找包在 `try/except DependencyUnavailableError` 中；后续 `cache.get_by_id()` 与 `cache.get_status()` 在同一路由里没有继续转换。`get_report()` 也在初始 try 后还有一次 `cache.get_status()`。
- Recommendation: 将整个 repository read sequence 包进同一个 dependency-boundary helper，或在 route 内统一把 `DependencyUnavailableError` 转为 `AppError(503, DEPENDENCY_UNAVAILABLE)`。

### [P2] FileCache 对 ownerless 条目的本地租户语义不够严格

- Location: [`src/ideago/cache/file_cache.py:107`](/Users/simonsun/github_project/IdeaGo/src/ideago/cache/file_cache.py:107), [`src/ideago/cache/file_cache.py:261`](/Users/simonsun/github_project/IdeaGo/src/ideago/cache/file_cache.py:261)
- Impact: 生产已禁止 FileCache，但开发和测试环境可能把 ownerless/orphan report 当作当前用户可访问，导致本地验证通过而 Supabase 行为不同。
- Evidence: `_get_by_id_sync()` 在提供 `user_id` 时，只拒绝 `entry.user_id` 存在且不同的情况；如果 entry 存在但 `user_id` 为空，仍继续读取 report 文件。`_delete_sync()` 也对 ownerless entry 不拒绝。
- Recommendation: 对受认证路径调用时要求 `entry.user_id == user_id`；匿名 claim/legacy migration 需要单独 helper，不要混入普通用户读取。

## 4. Likely Risks

### [P2] 多实例运行态与 admin metrics 仍可能误导排障

- Location: [`src/ideago/api/dependencies.py:45`](/Users/simonsun/github_project/IdeaGo/src/ideago/api/dependencies.py:45), [`src/ideago/api/routes/admin.py:194`](/Users/simonsun/github_project/IdeaGo/src/ideago/api/routes/admin.py:194)
- Impact: 多 worker 或多实例下，`ReportRunRegistry`、`PipelineTaskRegistry`、`app_metrics.snapshot()` 都只代表当前进程；运营人员可能看不到其他实例的活跃任务、失败率或请求量。
- Evidence: processing dedup 已接入 PG，但 SSE event history 和 task registry 仍是本进程对象；`/admin/metrics` 没有 instance id 或聚合说明。
- Recommendation: 先给 admin metrics 增加 `scope: instance` 与 instance id，再规划集中指标；对 report run visibility 提供 shared status 查询兜底。

### [P2] 测试集中度继续推高回归成本

- Location: [`tests/test_api.py`](/Users/simonsun/github_project/IdeaGo/tests/test_api.py), [`tests/test_langgraph_engine.py`](/Users/simonsun/github_project/IdeaGo/tests/test_langgraph_engine.py), [`pyproject.toml:187`](/Users/simonsun/github_project/IdeaGo/pyproject.toml:187)
- Impact: 小修复需要跑很重的测试上下文；定向 pytest 又会受全局 coverage 门槛影响，自动修复线程不容易快速给出可信反馈。
- Evidence: 当前最大测试文件分别为 5293 行和 3089 行；`pytest` 默认启用 `--cov-fail-under=80`。
- Recommendation: 增加不带 coverage gate 的 `pytest-smoke` 或 `pytest-targeted` 命令，并逐步按 route/service 拆分超大测试文件。
