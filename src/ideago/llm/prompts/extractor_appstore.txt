## Instructions
1. For each result that represents a real iOS app relevant to the user's intent, extract competitor information.
2. Prioritize apps that match the user's exact anchor ecosystem and intended relationship. If the user wants a tool built for a specific ecosystem, generic adjacent apps should not be treated as direct competitors.
3. Prioritize facts from `appstore_meta` (genre, rating, rating_count, price, version, release_date_iso). Treat `description` only as supporting context.
4. Do not copy marketing text verbatim into `one_liner`.
5. **CRITICAL: Only use URLs that appear in the provided data. NEVER fabricate or guess URLs.**
6. If details are missing, leave optional fields empty instead of guessing.
7. Assign relevance_score (0.0-1.0) using feature fit plus App Store signals (rating_count, rating, release freshness where available).
8. Also extract:
   - `pain_signals` (review pain clusters, severity/frequency)
   - `commercial_signals` (paid intent, conversion clues)
   - `migration_signals` (switch intent away from existing apps)
   - `evidence_items` with explicit categories
9. `evidence_items[].category` must be one of:
   - `competitor`, `pain`, `commercial`, `migration`, `whitespace`, `market`
10. Distinguish direct competitors from adjacent analogue apps.
    - Set competitor `relevance_kind` to exactly `direct` or `adjacent`.
    - Similar UX patterns alone do not make an app a direct competitor.
11. App review/blog/tutorial style pages usually work better as evidence, but if a page clearly points to or profiles a concrete app product relevant to the user's intent, you may still extract it as a competitor.
12. Write all natural-language fields in the specified `output_language`.
13. If `output_language` is `zh`, output Chinese only except unavoidable brand names / product names / URLs. If `output_language` is `en`, output English only except unavoidable brand names / product names / URLs.
14. For optional string fields, output `""` instead of `null`.
